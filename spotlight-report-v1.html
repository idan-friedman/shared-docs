design proposal · 2026-06-15

SpotlightReport

A unified, additive output schema both pipelines emit alongside their existing artifacts.

The deep-research and signal pipelines today produce two different output shapes for the same conceptual work-list. Consumers want one. Add one new top-level type that both pipelines emit — keeping every existing schema and on-disk artifact untouched.

slide 2 · TL;DR

What's changing

Five points that capture the proposal.
1
Additive, not replacing
One new top-level type — SpotlightReport. Both pipelines emit it next to existing outputs. SpotlightsResult, SignalPipelineResult, today's Candidate, prompts, and the renderer are all unchanged.
2
Flat top-level lists
candidates, findings, anomalies. No module_runs nesting. Each candidate/finding carries module_qualified_name as a string label.
3
One changes[] per candidate
DR's deep_research_proposals + agent_proposals fold into a single provenance-tagged changes list. Signal pipeline's stage-04 changes use the same shape.
4
Globally-unique ids
Adapter renumbers cand-NNNN / find-NNNN across the whole report; originals preserved on original_id / original_finding_id. source_refs rewritten consistently.
5
Two thin adapters at the output boundary
DR adapter: SpotlightsResult → SpotlightReport. Signal adapter: SignalPipelineResult + run-dir → SpotlightReport. Both pure (no I/O, no LLM, no prompt changes). New consumers read the report; old consumers ignore it.

Structural difference

Same work, different envelope. Today's per-module nesting collapses; module identity becomes a string label on each record.
Today — SpotlightsResult (DR)

├─ module_runs {
│ "v1.kv_offload"
│ ├─ candidates [
│ │ cand-0001

│ │ cand-0002  
│ │ cand-0003  …
│ └─ findings [find-0001…find-0011]
│ "v1.engine"

│ └─ findings [find-0001, …]

├─ extractor_invocation

Proposed — SpotlightReport
SpotlightReport

├─ candidates [

└─ issues

Renumbering — what the right pane shows
candidate id cand-0001 (module-scoped) → cand-0003 (global). Old id preserved on original_id; the chg-cand-NNNN-NN pattern picks up the new id in every embedded change.
finding ids find-0008..find-0011 (module-scoped) → find-0017..find-0020 (global). source_refs on each Change is rewritten to the new ids; old ids preserved on each ResultFinding.original_finding_id.
why these numbers Illustrative — assumes one earlier-walked module contributed 2 candidates and 9 findings. Actual offsets depend on alphabetical walk order across all modules in the run (open question Q1, slide 7).
slide 6 · adapters

Output-boundary adapters

One per pipeline. Pure: no I/O at the schema layer, no LLM calls, no prompt changes. Existing internal artifacts and result types stay exactly as they are.
Deep-research pipeline
SpotlightsResult
module_runs · per-module candidates with deep_research_proposals[] + agent_proposals[] · diagnostic envelopes
→
DR adapter
walk module_runs in deterministic order · renumber globally · merge proposal lists with provenance tags · rewrite source_refs to renumbered finding ids · sum cost/duration into RunInfo
→
SpotlightReport
written to <artifacts_dir>/spotlight_report.json — existing result.json unchanged
Signal pipeline
SignalPipelineResult + run-dir
01_signals/anomalies.json · 03_candidates/candidates.json · 04_changes/<cand-id>.json
→
Signal adapter
map Candidate.file → ProjectTree.modules[*] (deepest prefix wins; None if no match) · renumber · convert stage-04 Change → unified Change with source = telemetry_anomaly
→
SpotlightReport
written to runs/<id>/spotlight_report.json — all per-stage files unchanged
slide 7 · open

For tomorrow's walk-through

The spec locks most of §4. These three are still open and worth discussing live.
Q1 — renumbering
Deterministic id order across re-runs?
Spec recommendation: alphabetical by module_qualified_name, then in-module stored order. Two re-conversions of the same SpotlightsResult would then produce identical reports.
spec §4.3
Q2 — unmapped candidates
None over a sentinel string?
Signal candidates whose file matches no extracted module: module_qualified_name = None. Sentinel ("_unassigned") was rejected because it can collide with real names. Confirm.
spec §4.5
Q3 — relation to PR #A
Replace, supersede, or land alongside?
PR #A from the 2026-06-14 unification plan introduces a flat changes.json with candidate_ref. This proposal embeds Change and drops candidate_ref.
spec §4.4
